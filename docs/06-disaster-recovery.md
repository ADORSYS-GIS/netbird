# 06 - Disaster Recovery

This document outlines the backup and recovery procedures for the NetBird infrastructure.

## Backup Strategy

### Database (PostgreSQL)
*   **Automatic Backups**:
    *   **AWS RDS**: Automated snapshots enabled by default (7 days retention).
    *   **GCP Cloud SQL**: Automated backups enabled (7 days).
    *   **Azure Database**: Automated backups enabled (7 days).
*   **Manual Snapshots**:
    To take a manual snapshot before major changes:
    *   **AWS**: `aws rds create-db-snapshot ...`
    *   **GCP**: `gcloud sql backups create ...`

### Configuration State
*   **Terraform State**: Stored remotely (S3/GRS/Azure Blob). **CRITICAL**: Do not lose access to this bucket.
*   **Ansible Inventory**: Generated by Terraform, can be re-generated anytime.
*   **Secrets**: Stored in Vault or Terraform Variables. Ensure you have a copy of `terraform.tfvars`.

## Restore Procedures

### Database Point-in-Time Recovery (PITR)
If data corruption occurs (e.g., accidental deletion):

1.  **Identify Timestamp**: Determine the exact time before the corruption.
2.  **Restore to New Instance**:
    *   Use Cloud Console or CLI to restore to a *new* DB instance.
    *   Do **NOT** overwrite the existing integration; create a new one to verify data first.
3.  **Update Terraform**:
    *   Point `main.tf` (database module) to the new instance ID? No, Terraform manages the instance lifecycle.
    *   Better: Use Terraform `restore_to_point_in_time` block if supported by module, OR rename the new instance to the old name after deleting the old one.

### Full Infrastructure Redeployment
If a region fails or critical state is lost:

1.  **Modify Region**: Update `aws_region` / `gcp_region` in `terraform.tfvars`.
2.  **Run Terraform**: `terraform apply` will provision a fresh environment in the new region.
3.  **Restore Data**: Import the database dump/semester snapshot to the new DB instance.
4.  **Update DNS**: Point `netbird_domain` to the new Reverse Proxy IP.
5.  **Run Ansible**: Configure the new VMs.
